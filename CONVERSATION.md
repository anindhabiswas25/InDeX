## 1. What is Staking?

Staking means **locking up your tokens to help secure a blockchain network**,
in exchange for rewards.

It is the Proof of Stake equivalent of Bitcoin mining. Instead of spending
electricity to secure the network, you spend capital — you lock up tokens.

**Who does it?**

- **Validators** — Run actual server nodes 24/7. Need technical expertise.
  Put up their own INIT as collateral. Examples: Chorus One, TrustedPoint.
- **Delegators** — Ordinary token holders. No technical knowledge needed.
  Just lock their tokens and point them at a validator.

**Why do people stake?**

Simple passive income on tokens they already hold.

```
Hold 1000 INIT, do nothing  → still 1000 INIT next year
Stake 1000 INIT at 15% APY  → 1150 INIT next year
```

There is no reason NOT to stake if you are holding long term.

## 4. What is a Block?

A block is a **batch of transactions grouped together and recorded permanently**
on the blockchain.

Think of it like pages in a ledger:
- Page 1 (Block 1): "Alice sent 10 INIT to Bob, Charlie staked 100 INIT..."
- Page 2 (Block 2): "Dave borrowed 50 INIT, Eve swapped tokens..."
- Every ~2 seconds on Initia L1, a new page is written forever.

---

## 5. What Does a Validator Actually Do?

Validators are the **nodes that write those blocks**.

The process:
1. Users broadcast transactions to the network
2. Validators collect pending transactions from the mempool
3. One validator is randomly selected (weighted by stake) to **propose** the next block
4. Other validators **vote** to confirm the block is valid
5. Once 2/3 of validators agree → block is finalized → permanently written
6. The proposer + all validators who voted get rewarded

**Validators are accountants who verify every transaction is legitimate:**
- Is this signature valid?
- Does this person have enough balance?
- Is this smart contract execution valid?
- Is anyone trying to double spend?

If a validator approves a fraudulent block → they get **slashed** (lose a % of
all delegated INIT including yours).

## 6. What Are You Actually Giving a Validator?

**You are giving the validator nothing physical.**

You are giving them **trust and economic weight**.

When you send MsgDelegate:

```
You sign a message:
"I want my 1000 INIT to count toward Chorus One's voting power"

→ Message goes to L1 staking module
→ Staking module records: Chorus One total stake += 1000 INIT
→ Your 1000 INIT NEVER moves to Chorus One's wallet
→ Chorus One sees their voting weight increased
```

**What you actually give:**

1. **Voting weight** — your stake increases their chance of being selected
   to propose a block
2. **Economic security** — your INIT is collateral. If Chorus One misbehaves
   and gets slashed, you lose your stake too. By delegating you are co-signing
   their promise to behave honestly.

**Court system analogy:**

```
Judge (Validator) must post a BOND before they can judge cases
If judge makes corrupt decisions → bond confiscated (slashing)

You as delegator:
  Co-sign the judge's bond
  Your money is at risk if judge is corrupt
  In return you share in the judge's salary (block rewards)
  Validator takes commission; you get the rest
```

**The reward is payment from the Initia protocol for providing economic
security — not payment from the validator out of their pocket.**

## 7. Where Does Your INIT Go When You Delegate?

Your INIT goes into the **L1 Staking Module** — a protocol-owned vault managed
by the Initia protocol itself, not by any validator.

```
Your Wallet
"Delegate 1000 INIT to Chorus One"
        ↓
L1 Staking Module (protocol vault)
"Recorded: init1j55... has 1000 INIT backing Chorus One"
        ↓
Chorus One voting power increases by 1000 INIT
Chorus One NEVER receives or touches your INIT
```

**The validator receives:**
- Increased voting power (recorded by staking module)
- Commission on rewards your delegation generates (newly minted INIT)
- More block proposal opportunities

**Shareholder analogy:**
- You buy 1000 Apple shares → shares go into central depository (staking module)
- CEO gets more voting authority (validator gets more weight)
- Company pays dividends (protocol mints rewards)
- CEO takes management fee (validator commission)
- Your shares never left the depository — always yours to take back

**Getting INIT back (undelegation):**
- Call MsgUndelegate
- Unbonding period: 7 days (testnet), 21 days (mainnet)
- During unbonding: earns no rewards
- After unbonding: INIT returns directly to your wallet from staking module
- Validator has no say — fully automated by protocol

**Why unbonding period exists:**
Security mechanism. If validators could be undelegated instantly, they could
misbehave and delegators could escape before slashing is processed.

## 8. How Does the Exchange Rate Work?

The exchange rate is one formula in the staking contract:

```rust
rate = total_init_staked * 1_000_000 / total_initx_supply
// 1.0 = 1_000_000
// 1.09 = 1_090_000
```

**At genesis (protocol starts at zero):**
```
total_init_staked  = 0
total_initx_supply = 0
rate = hardcoded 1.0 (genesis default)
```

**First deposit sets the baseline:**
```
User deposits 1000 INIT
Contract mints 1:1 → 1000 INITx

total_init_staked  = 1000
total_initx_supply = 1000
rate = 1.0
```

**After keeper calls AddRewards(100 INIT):**
```
Protocol fee (10%) = 10 INIT → treasury
Staker reward (90%) = 90 INIT → total_init_staked

total_init_staked  = 1090  ← went up
total_initx_supply = 1000  ← unchanged
rate = 1090/1000 = 1.09
```

**What changes total_init_staked:**
- UP: user deposits INIT
- UP: keeper calls AddRewards (yield from fees + L1 staking)
- DOWN: user withdraws (burns INITx)
- DOWN: validator slashing

**What changes total_initx_supply:**
- UP: user deposits INIT → new INITx minted
- DOWN: user withdraws → INITx burned

**What does NOT affect exchange rate:**
- LP pool swaps
- Lending activity
- INITx transfers between wallets
- INITx used as collateral
- INITx provided to LP pool

**Why getting fewer INITx is NOT a loss:**
```
Person A staked early at rate 1.0:
  10 INIT → 10 INITx
  Withdraws at rate 1.12: 10 × 1.12 = 11.2 INIT ✅ profit

Person B staked late at rate 1.12:
  10 INIT → 8.93 INITx
  Withdraws at rate 1.15: 8.93 × 1.15 = 10.27 INIT ✅ profit
```

Getting fewer INITx at a higher rate means each INITx is worth more INIT.
The math always gives you back what you put in plus accumulated yield.

**Mutual fund analogy:**
```
Fund starts:   price per unit = ₹10, buy 10 units for ₹100
Fund grows:    price per unit = ₹15, new investor buys ₹100 → gets 6.67 units
Later:         price = ₹20
You withdraw:  10 × ₹20 = ₹200 ✅ doubled
New investor:  6.67 × ₹20 = ₹133 ✅ also profit
```

## 9. How Does the LP Swap Rate Work?

The LP pool uses the **constant product formula** (same as Uniswap v2):

```
x × y = k  (always constant)
x = INIT reserve
y = INITx reserve
k = constant product
```

**Swap rate = init_reserve / initx_reserve**

Changes every single swap. Nobody controls it — it is emergent from trading activity.

**Example swap (INIT → INITx):**
```
Pool: 10,000 INIT reserve, 10,000 INITx reserve
User swaps 1,000 INIT

Step 1: Fee deducted (0.3%)
  fee = 1000 × 0.3% = 3 INIT
  input after fee = 997 INIT

Step 2: AMM formula
  out = (10,000 × 997) / (10,000 + 997) = 907 INITx

Step 3: Protocol fee from the 3 INIT fee
  protocol_fee ≈ 0.05% of input → stored in accrued_fees
  keeper harvests this → feeds into staking exchange rate

New pool: 11,000 INIT, 9,093 INITx
New rate: 11,000 / 9,093 = 1.21
```

**Key difference from staking exchange rate:**

| | Staking Exchange Rate | LP Swap Rate |
|---|---|---|
| Set by | Protocol math | Market (pool reserves) |
| Changes when | Keeper adds rewards | Every swap |
| Direction | Only up (normally) | Up and down |
| Controlled by | Nobody — automatic | Traders indirectly |

**Indirect connection to exchange rate:**
```
LP swap → 0.05% protocol fee accrues
→ Keeper harvests fees every 10 min
→ Keeper calls AddRewards
→ total_init_staked increases
→ Exchange rate goes up
```

LP activity indirectly increases the staking exchange rate through fees.
More trading volume = more fees = faster rate growth.

## 10. Are Staking, Swapping, Liquidity, and Lending Separate?

**Yes. They are completely separate contracts.**

```
STAKING CONTRACT     LP POOL CONTRACT     LENDING CONTRACT
─────────────────    ────────────────     ────────────────
Deposit INIT         Swap INIT ↔ INITx    Supply INIT
Get INITx            Add/Remove           Borrow INIT against
Rate appreciates     Liquidity            INITx collateral
from yield           Earn swap fees       Fixed 5% APR
```

**They connect only through fees flowing to the keeper:**
```
LP swap fees      → keeper harvests → AddRewards → rate goes up
Lending fees      → keeper harvests → AddRewards → rate goes up
```

**The beautiful design:** A person who just stakes INIT and holds INITx benefits
from ALL activity in LP and Lending — without doing anything or taking on any
of that risk. Every trade, every loan generates fees that flow into the staker's
exchange rate.

**Who uses what:**

| Trader | Swapping only | Get tokens at market price |

## 11. How Does Total INITx Supply Stay the Same?

When a user uses INITx for swapping, LP, or lending — **total supply does not change**.

INITx tokens just move between wallets (addresses). They are not created or destroyed.

```
Total INITx in existence = 1000

User swaps 100 INITx on LP pool:
  Before: user wallet has 100, others have 900
  After:  LP pool has 100, others have 900
  Total:  still 1000 ← tokens just changed address

User provides 100 INITx as LP liquidity:
  Before: user wallet has 100
  After:  LP pool contract has 100
  Total:  still 1000 ← moved wallets

User locks 100 INITx as lending collateral:
  Before: user wallet has 100
  After:  lending contract has 100
  Total:  still 1000 ← moved wallets
```

**Total supply ONLY changes when:**
- Staking contract MINTS INITx (on deposit) → supply increases
- Staking contract BURNS INITx (on withdrawal) → supply decreases

**Movie ticket analogy:**
- Cinema prints 1000 tickets → total = 1000
- Ticket moves hand to hand → total stays 1000
- Only changes when cinema prints more OR destroys used ones
- Staking contract is the cinema. Mint = print. Burn = destroy.

## 14. Is This the Same on Every Blockchain?

The core concept is universal across all Proof of Stake chains. But important
differences exist:

**What is the same everywhere:**
- Validators produce and verify blocks ✅
- Slashing if validator misbehaves ✅
- Unbonding period when undelegating ✅
- Delegators share in block rewards ✅

**Key differences by chain:**

| Chain | Difference |
|---|---|
| Ethereum | Need 32 ETH to be validator. No native delegation — use Lido etc. |
| Solana | No unbonding period (instant). Rewards per epoch (~2-3 days). |
| Cosmos chains | Identical to Initia BUT standard x/staking module |
| Polkadot | Called "nomination". Up to 16 validators. Rewards per era (~24h). |
| Bitcoin | Proof of Work, not Proof of Stake. No delegation concept. |

**What makes Initia special vs standard Cosmos:**
```
Standard Cosmos: only native token can be delegated (e.g. ATOM)
Initia mstaking: INIT + Move LP tokens can BOTH be delegated
                 Multiple assets secure the network simultaneously
                 Called "enshrined liquidity" — Initia's key innovation
```

Initia uses `/initia.mstaking.v1.MsgDelegate` (NOT standard cosmos staking).
The `amount` field is a **repeated array of coins**, not a single coin.

## 17. How Does the Protocol Send INIT to a Validator Over Time?

Users deposit at different times in different amounts. The protocol handles this
by **pooling and batching**.

```
Day 1: User A deposits  100 INIT ┐
Day 3: User B deposits  500 INIT ├→ All goes into staking contract vault
Day 7: User C deposits   50 INIT │  total_init_staked = 1,650 INIT
Day 9: User D deposits 1000 INIT ┘

Keeper bot runs:
  Total in vault = 1,650 INIT
  Buffer (20%)   =   330 INIT → stays on wasm-1
  Surplus (80%)  = 1,320 INIT → bridge to L1

Keeper bridges 1,320 INIT → initiation-2
Keeper sends ONE MsgDelegate: 1,320 INIT → Chorus One

Chorus One sees ONE delegator (keeper wallet)
They have no idea it came from 4 different users
```

**Why batching is better than per-user delegation:**

```
Per-user: each deposit = bridge tx + delegate tx = high gas, complex
Batching: all deposits → keeper runs once → one bridge + one delegate
          Much cheaper, simpler, more reliable
```

**Over time as more deposits come in:**
```
Week 1: Delegate 1,320 INIT to Chorus One
Week 2: 2,000 more INIT deposited → delegate 1,600 more INIT
Week 3: 500 more deposited → delegate 400 more INIT

Chorus One sees growing delegation from one address
Total delegated: 0 → 1,320 → 2,920 → 3,320 INIT

Rewards flow back every harvest:
  Keeper claims uinit on L1
  Bridges back to wasm-1
  AddRewards called
  Exchange rate goes up for EVERYONE
```

## 19. Full Withdrawal Flow

### Small Withdrawal — Instant (Buffer Covers It)

```
User calls Unstake(500 INITx)
        ↓
Contract checks liquidity buffer: 2,000 INIT available ✅
        ↓
Contract burns 500 INITx
        ↓
INIT owed = 500 × exchange rate 1.05 = 525 INIT
        ↓
Contract sends 525 INIT directly to user from buffer
        ↓
Done. Instant. No waiting.
```

### Large Withdrawal — Queued (Buffer Cannot Cover It)

```
User calls Unstake(5,000 INITx)
        ↓
Buffer only has 2,000 INIT — cannot cover ❌
        ↓
Contract burns 5,000 INITx immediately
(user no longer earns yield on this amount)
        ↓
Withdrawal queued: "user wants 5,250 INIT"
        ↓
Keeper detects pending queue
        ↓
Keeper calls MsgUndelegate on L1 for 5,250 uinit
        ↓
L1 unbonding period: 7 days (testnet)
        ↓
After 7 days → INIT released from staking module to keeper wallet
        ↓
Keeper bridges INIT: L1 → wasm-1 via IBC
        ↓
Keeper calls FulfillWithdrawal on contract
        ↓
User receives INIT. Total wait: ~7 days + bridge time
```

### Bypass Queue — Instant via LP Pool

```
User doesn't want to wait 7 days
        ↓
User swaps INITx → INIT on LP pool
        ↓
Gets INIT instantly at market price
        ↓
Protocol earns swap fee → feeds back into exchange rate
```

| Mechanism | Formula |
